#### WARNING-3 ✅ RESOLVED: Web spec says "EventSource"; code uses `fetch + ReadableStream`

### SUGGESTION (nice to fix)

#### SUGGESTION-1: Default provider ping-fail does not fail startup
- **Spec**: `multi-llm-provider-support/spec.md` lines 35-38, 45-48 — "全部 provider 失败 MUST 启动失败" and "默认失败 MUST 启动失败"
- **Implementation**: `cmd/server/main.go:52` calls `PingAll` and populates `Health` map, but never `os.Exit(1)` on failure. `pickDefaultClient` (`main.go:197`) returns `nil` if default fails; `runnerFactory` then dials `nil` per chat request, producing a runtime SSE error.
- **Impact**: User starts the server with no working LLM and gets confusing runtime errors per chat turn, rather than a clear startup failure.
- **Recommendation**: Add a startup check in `main.go` after `PingAll` — if `Health[cfg.LLM.Default].OK == false`, `os.Exit(1)` with stderr "default provider <name> is unreachable". Same for the "all providers failed" case (low priority — partial success is fine).

#### SUGGESTION-2: UI backup warning is in README, not as a first-start UI prompt
- **Spec**: `k8s-credential-encryption/spec.md` line 58 — "UI MUST 提示 'master.key 与 data.db 必须一起备份...'"
- **Implementation**: Warning is in `README.md` (lines 22-24) and a `docs/` link. No first-start UI prompt in the React app.
- **Impact**: First-time users may not see the warning if they skip the README. The backup constraint is the project's biggest "data loss" risk per `design.md` R4.
- **Recommendation**: Add a one-time banner/toast on the chat view's first mount that shows the backup warning and dismisses permanently (localStorage flag). Quick add to `web/src/App.tsx` or `ChatView.tsx`.

#### SUGGESTION-3: `k8s-credential-encryption` "首次启动提示" not enforced as code
- **Spec**: Same as SUGGESTION-2.
- **Implementation**: No code path triggers the UI prompt.
- **Recommendation**: Same as SUGGESTION-2.

#### SUGGESTION-4: `audit_log.action` taxonomy not documented
- **Spec**: `k8s-write-with-plan-preview/spec.md` line 65 + `k8s-policy-guardrails/spec.md` line 83-87 require audit for plan execution + policy changes with structured fields. The audit log schema (`internal/store/audit.go:38`) has session_id, cluster_id, action, target, status, message — but the set of `action` values (e.g. `plan_execute`, `policy_update`, `policy_change`) is not enumerated anywhere.
- **Recommendation**: Document the audit action vocabulary in `docs/` (could be a small `docs/audit-actions.md` or appendix to `docs/default-policies.md`). This makes it easy for users to write SQL queries against the audit log.

## Drift Summary

5 plan-time decisions were revised during implementation. All were improvements with commit-time documentation:

1. **Operation JSON bridge** (commit `e8f1dfe`): Plan had unexported fields; encoding/json silently bypassed policy. Fix: `operationWire` bridge struct.
2. **UnsafeFields OR semantics** (Task 6): Plan implied AND; implementation uses OR (any one match → rule fires) because AND would make rules effectively unusable. Documented in `internal/policy/engine.go` comment.
3. **`PingAll` wiring** (commit `233c119`): Plan had it in `cmd/server/main.go` but the implementer left it as a no-op. Caught by `/opsx:verify`, fixed.
4. **SSE client uses fetch** (Task 10): Plan said `EventSource`; the implementer correctly used `fetch + ReadableStream` because the backend is POST. Spec is stale on this point (see WARNING-3).
5. **RunnerFactory** (Task 13): Plan had it as nil in main.go; wiring was deferred to Task 13. Final shape is a closure matching `server.RunnerFactory`.
